KEY FEATURES
Built for production-scale task execution
  • SQLite-based queue with atomic operations prevents race conditions
  • Parallel or serial execution - N workers configurable (serial when N=1)
  • Real-time web dashboard with WebSocket updates at localhost:8765
  • Process isolation - each worker is separate, no shared state
  • Automatic retries with configurable max attempts per task
  • Orphan detection - stuck tasks auto-reset after 30 minutes
  • Graceful shutdown - workers finish current task (5min timeout)
  • Emergency stop - immediate termination when needed

TECHNICAL DETAILS
How it works under the hood
SQLite IMMEDIATE
Transaction isolation prevents race conditions. Only one worker can claim each task.
Multiprocessing
Each worker is a separate Python process. Full isolation, independent failure domains.
FastAPI + WebSockets
Dashboard backend pushes real-time updates. No polling, instant feedback.
Session Model
Each task spawns TWO Amplifier sessions: builder (implements) + validator (verifies). Full subprocess isolation.
Heartbeat Monitoring
Workers send heartbeat every 30s. Orchestrator detects crashes and restarts workers automatically.
USE CASES
When to use amplifier-swarm
  • Large implementation projects - 50+ tasks to complete across a codebase
  • Testing serial first, then scaling - Start with N=1, increase when confident
  • Long-running workflows - Multi-hour processes that need monitoring
  • Resumable work - Interrupted? Just restart, it picks up where it left off
  • Multi-phase projects - Import tasks by phase, execute in waves
Real Example: WebWord Implementation
50 implementation tasks, 85.5 estimated hours. With 3 workers: completed in ~30 hours of wall-clock time. Manual oversight via dashboard prevented "is it stuck?" anxiety.
